GROUND RULES

Content Storage — Choose One
Format What it is Pros Cons Verdict
TOML Config file format — like a readable INI file. [section] headings, key = value pairs. Used by pyproject.toml, Cargo.toml. Human-readable, diff-friendly, designed for config. Claude generates it cleanly. Python 3.11+ reads it natively. Unfamiliar to most devs. No querying. Files are the "rows". RECOMMENDED for content
JSON JavaScript Object Notation — universal text format. Every language reads it. More familiar than TOML. Universal familiarity. Python reads/writes natively. Easy to validate with Pydantic. No comments. More verbose than TOML. Harder to hand-edit (brackets/commas). FINE ALTERNATIVE — same idea, different syntax
SQLite Embedded SQL database — a real database in a single file. No server. Built into Python (sqlite3). Used by browsers, phones, most apps internally. Real SQL queries. Familiar "database" concept. Good for complex relational data. Overkill for static content. Requires SQL schema + migrations. Content authoring needs INSERT statements or a GUI tool. SKIP FOR CONTENT — use for save state only if JSON gets complex
Decision: TOML for all content files (scientists, anomalies, experiments, containment, narrative templates). JSON for the runtime save file. No SQL database needed — the "database" is just a folder of text files, one file per entity. Think of it like a mini CMS that Claude can generate and you can hand-edit.
Pydantic for Schema Validation

Every TOML entity has a corresponding Pydantic model. Loading = parse TOML → validate with Pydantic → in-memory Python object. Content errors (missing fields, wrong types) surface at startup, not mid-game. Think of Pydantic as the type-checker for your content files.

Runtime State — Encapsulation Pattern
PATTERN

Yes — this is the Encapsulation pattern (OOP fundamentals). GameState's internal attributes are private (_ledger, _scientists). All reads and writes go through public methods. This means:

class GameState: # Private — no direct access from outside _ledger: Ledger _scientists: dict[str, ScientistState] _history: list[RoundRecord] # Public read interface def get_scientist(self, id) -> ScientistState: ... def get_active_scientists(self) -> list: ... def get_tech_level(self) -> int: ... # Public write interface — controlled mutations def apply_outcome(self, result: RoundResult): ... def tick_pressure(self, delta: int): ... # Humanity is NEVER exposed as a raw number def is_humanity_critical(self) -> bool: ...

Humanity and the hidden ledger values are never returned raw — only methods like is_humanity_critical() and get_tone_modifier() expose derived signals. This enforces the game's design contract at the code level.

Classification Color Convention
SAFE Low threat, predictable. Low pressure per round, higher tech gain. Good for damaged scientists.
EUCLID Moderate threat, somewhat unpredictable. Balanced risk/reward. Most anomalies are EUCLID.
KETER High threat, severe pressure. Best tech gain on success, catastrophic failure cost. Forces hard choices.
MVP distribution: 2 SAFE · 3 EUCLID · 1 KETER. The KETER appears in rounds 5–6 of a typical run. Roguelike shuffle means players don't always reach it.

Resolution Formula
# Called by ResolutionEngine each round def resolve_experiment(scientist, experiment, state): # Base roll: 0-100 roll = random.randint(0, 100) # Scientist contributes their relevant skill skill_bonus = scientist.skills.containment * 2 # Hidden humanity penalty (player never sees this) humanity_penalty = max(0, (50 - state.ledger.humanity) // 5) # Tech bonus for unlocked experiments tech_bonus = state.ledger.tech_level // 10 final = (roll + skill_bonus + experiment.roll.modifier + tech_bonus - humanity_penalty) success = final >= experiment.roll.base_difficulty return success, final, roll # only success shown
Player signal: Only success = True/False is shown. The narrative text describes how close it felt. Low humanity runs "feel" harder without the player seeing a penalty.

Containment vs. Experiment — Design Note

Experiments generate knowledge and carry personnel risk. Containment converts that knowledge into safety — reducing anomaly pressure. The order matters: you pick an experiment (risk, gain), then separately pick containment (cost, pressure reduction). Both decisions happen in the same round.

Containment does NOT have a success/failure roll — it always works if selected. The risk was already borne by the experiment. This keeps the decision tree clean: one roll per round, not two.

Tech unlock pattern: Basic containment is always available. Better containment unlocks as tech_level rises. This is how the "tech corpus" resource manifests as visible game options rather than hidden numbers.

KEY PACKAGES

Toolchain: Astral stack. uv replaces pip + virtualenv + pip-tools — one fast tool for installs, lockfiles, and scripts. ruff replaces flake8 + isort + black — linting and formatting in one pass. ty is Astral's new type checker (replaces mypy). All three are Rust-native, 10–100× faster than the Python equivalents. Run: uv init the_protocol · uv add textual pydantic loguru platformdirs · uv run ruff check . · uv run ty check .

Tier 1 = essential core · Tier 2 = important but swappable · Tier 3 = tooling/dev only

textual TIER 1
TUI framework. Widgets, CSS layout, reactive state, keyboard/mouse events. The entire game UI is built in Textual. pip install textual
Key classes: App, Widget, ScrollableContainer, RichLog, Static
pydantic v2 TIER 1
Data validation and serialization for all content schemas and game state. Catches malformed TOML at load time. Fast Rust core in v2.
pip install pydantic — use model_validate() from TOML dicts
tomllib (stdlib) TIER 1
Built into Python 3.11+. Zero install cost. Parses TOML → Python dict → feed to Pydantic. For write support (saving game config) use tomli-w.
import tomllib — no pip needed. Require Python 3.11+ in pyproject.toml
rich TIER 1
Textual depends on Rich internally. Use directly for Text objects with per-character color styling — critical for coloring scientist names and art inline.
Installed as Textual dep. Use Text.from_markup() for inline colored spans
steamworks.py TIER 2
Python bindings for the Steamworks SDK. Achievements, cloud saves, overlay, stats. Wrap in a SteamManager class that gracefully no-ops when Steam isn't running (dev mode).
Requires Steamworks SDK + steam_api DLL alongside binary. Init in a try/except.
pyinstaller TIER 2
Bundles Python app + interpreter into a single folder (or --onefile). The standard for Textual apps on Steam. Use --collect-data textual to include CSS assets.
Alternatively: Nuitka compiles to C — smaller binary, no interpreter, but slower build cycle
loguru TIER 2
Drop-in logging with structured output. Log resolution engine decisions, content load errors, and Steam SDK failures to ~/.foundationprotocol/logs/. Invaluable for debugging shipped builds.
pip install loguru — use logger.bind(round=n) for context
platformdirs TIER 2
Cross-platform user data directories. user_data_dir("foundationprotocol") returns the right path on Windows (%APPDATA%), Mac (~/Library), Linux (~/.local/share).
pip install platformdirs — use for saves, logs, config
pytest + pytest-asyncio TIER 3
Testing the resolution engine, content loader, and game state serialization. Textual has a built-in App.run_test() for widget testing. pytest-asyncio needed since Textual is async.

REMAINING ARCHITECTURE DECISIONS

These need to be resolved before or at the start of implementation — they affect the shape of the code, not just content.

ARCH-01 Narrative template engine: string interpolation — confirmed
RESOLVED

Use Python's built-in str.format_map(context) for all narrative templates. Templates in TOML look like: "{{scientist.name}} {{experiment.verb_past}} the {{anomaly.descriptor}} entity." The NarrativeEngine.render(template, context) method passes a flat dict. Simple, zero deps, easy for Claude to generate template strings. Upgrade to Jinja2 only if conditional logic is needed in templates (e.g., different text if scientist is damaged).

ARCH-02 Content loading: eager at startup — confirmed
RESOLVED

All TOML content files load into a ContentRegistry singleton when the app starts. <1 second for MVP scope. Fail fast if any file is malformed — content errors surface before the player sees anything.

ARCH-03 Scientist color: both art panel and surrounding text — confirmed
RESOLVED

The scientist's hex color is applied to both: their ASCII art panel (via Textual widget color CSS) and all text attributed to them in the log (name, dialogue, incident reports). A single Textual Static widget with inline style="color: {scientist.color}" covers both. Their color is their identity — it persists in the scroll history even after they're gone.

ARCH-04 Narrative pool selection: no-repeat queue — confirmed
RESOLVED

Each narrative pool uses a shuffle-and-exhaust queue. The pool is shuffled at run start; lines are drawn in order and never repeat until the pool is exhausted, then reshuffled. Zero repetition risk for MVP pool sizes. Stored in GameState._narrative_queues: dict[str, deque[str]].

ARCH-05 Window mode: self-contained window — confirmed
RESOLVED

The game launches in its own self-contained terminal window — not in the user's existing terminal. For Steam on Windows/Mac/Linux, Textual apps can be launched via Windows Terminal, xterm, or kitty as subprocesses. The Steam launch command becomes wt.exe python main.py (Windows) or xterm -e python main.py (Linux). This gives us a fixed, controlled viewport with no dependency on the user's terminal setup.

Art column width: 44 chars. Minimum window: 80×24. Textual can enforce this with a startup size check.

TESTING STRATEGY

Three layers: isolated unit tests, end-to-end integration tests, and autonomous agent playthroughs with recorded output posted to GitHub.

TEST-00 Startup Smoke Test — load all content through Pydantic, fail fast on schema breaks

Runs the full content load pipeline against real data files. If any TOML file is malformed or violates the Pydantic schema, this test catches it immediately. The first test to run in CI — if it fails, nothing else matters.

def test_content_registry_loads_all(): """If this fails, a content file broke the Pydantic schema.""" registry = ContentRegistry.load_from_dir("data/") # Every entity loaded without raising assert len(registry.scientists) == 3 assert len(registry.anomalies) == 6 assert len(registry.experiments) == 15 # All cross-references resolve (no dangling IDs) for anomaly in registry.anomalies.values(): for exp_id in anomaly.experiments: assert exp_id in registry.experiments, \ f"Anomaly {anomaly.id} references unknown experiment {exp_id}" # All art files exist on disk for sci in registry.scientists.values(): for art_path in [sci.art.intact, sci.art.damaged, sci.art.corrupted]: assert Path(art_path).exists(), f"Missing art: {art_path}"
Add test_content_registry_loads_all() to every CI run. As the data/ directory grows, this is the only guard against broken content breaking the game silently.
TEST-01 Unit Tests — pytest.mark.parametrize for critical isolated logic

Fast, isolated tests for the resolution engine, ledger mutations, and save/load round-trips. Use parametrize to cover edge cases without boilerplate.

import pytest from engine.resolution import ResolutionEngine # Test that high humanity_penalty correctly depresses rolls @pytest.mark.parametrize("humanity,expected_max_roll", [ (100, 110), # full humanity → no penalty (50, 100), # mid → small penalty (0, 80), # zero humanity → -30 penalty applied ]) def test_humanity_penalty(humanity, expected_max_roll): state = make_test_state(humanity=humanity) rolls = [ResolutionEngine._compute_roll(state, scientist, exp) for _ in range(100)] assert max(rolls) <= expected_max_roll # Test scientist integrity thresholds map to correct art state @pytest.mark.parametrize("integrity,expected_art", [ (100, "intact"), (70, "intact"), (69, "damaged"), (35, "damaged"), (34, "corrupted"), (1, "corrupted"), (0, "absent"), ]) def test_art_state_thresholds(integrity, expected_art): sci_state = ScientistState(integrity=integrity) assert sci_state.art_key == expected_art
TEST-02 Integration Tests — full round and run simulations

A small number of tests that run the full pipeline: load real content files → create game state → simulate a round → assert ledger changes and narrative output. Catches schema/validation regressions.

def test_full_round_pipeline(): # Load real content files registry = ContentRegistry.load_from_dir("data/") state = GameState.new_run(registry) # Simulate a complete round anomaly = registry.get_anomaly("scp_031") scientist = registry.get_scientist("morrow") experiment = registry.get_experiment("resonance_test") containment = registry.get_containment("resonance_cage") result = ResolutionEngine.resolve( state, anomaly, scientist, experiment, containment ) # Assert ledger changed (direction, not exact values) assert state.get_tech_level() > 0 assert result.narrative_text != "" assert state.get_round() == 2 def test_full_run_reaches_end_condition(): # Simulate a run to completion (forced bad choices) state = run_to_completion(strategy="always_risky") assert state.get_end_condition() in [ "containment_failure", "transformation", "full_containment" ]
