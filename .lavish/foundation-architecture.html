WHAT ARE WE BUILDING?

TOP SECRET

A round-based, text-and-ASCII-art-driven strategy game set inside a clandestine anomaly containment foundation. The player is the Director — every round a new anomaly arrives and demands three decisions: assign a Lead Scientist, select an experiment, choose a containment approach. The game's central tension is hidden: you are always trading human cost for institutional power, but the ledger is deliberately obscured.

Core Loop
Anomaly → Assign → Experiment → Contain → Report → Repeat
Core Feeling
Humanity slowly traded for safety & knowledge
Release Target

A — Rust + Ratatui

AUTHENTIC TERMINAL

Build the game as a true terminal application using Ratatui (TUI framework) + Crossterm (cross-platform terminal control). Steamworks via steamworks-rs. Ship as a standalone binary.

+ Tiny binary, zero runtime deps
+ True terminal experience — runs in Steam's overlay or a real terminal
+ Excellent for ASCII art rendering and color control
− Steep learning curve, slow iteration
− Ratatui has no built-in animation system
~ Steam: package executable + steam_appid.txt, use steamworks-rs crate
Key Crates
ratatui crossterm steamworks-rs serde_json rand

B — Python + Textual

FASTEST ITERATION

Textual by Textualize gives you a rich TUI framework with CSS-like styling, animations, and color gradients. PyInstaller or Nuitka bundles the interpreter. Steamworks via steamworks.py.

+ Rapid prototyping — high content velocity
+ Rich color system, gradients, animations built-in
+ Easy scripting for narrative content
− Bundled binary is 60–100 MB
− Startup time with PyInstaller can be slow
~ Steam: PyInstaller + steamworks.py, works on all 3 platforms
Key Packages
textual rich steamworks.py pydantic nuitka

C — Tauri + Xterm.js

BEST OF BOTH

Tauri wraps web content in a native Rust shell. Use Xterm.js to simulate a terminal, or fully custom HTML/CSS for the "terminal UI" with complete control over rendering. Steamworks via Tauri plugin.

+ Full control over visuals — pixel-perfect CRT effects, glows
+ Web ecosystem: animations, fonts, SVG art
+ Small binary (~10 MB) compared to Electron
− "Fake" terminal feel — purists may notice
− Tauri Steam integration requires custom setup
~ Steam: tauri-plugin-steamworks, package as .exe/.app
Key Tech
tauri 2.x xterm.js react/svelte tauri-plugin-steamworks

D — Godot 4

EASIEST STEAM

Use Godot 4 with a fully custom terminal-style UI (Label nodes, custom fonts, shader effects for CRT scanlines). GodotSteam is the most battle-tested Steamworks integration in indie dev.

+ GodotSteam: achievements, cloud saves, overlay all work out of the box
+ Built-in animation, audio, and scene management
+ GDScript is fast to write narrative content in
− Overkill for a text game; engine overhead
− Terminal feel requires deliberate shader work
~ Steam: GodotSteam plugin, export templates, ~70 MB
Key Tech
Godot 4 GodotSteam GDScript CRT shader

E — TypeScript + Electron

WEB ECOSYSTEM

React or Svelte frontend rendered in Electron with complete freedom to design a pixel-perfect terminal aesthetic. greenworks or steamworks.js for Steam integration. Ink if you prefer a React-for-CLIs approach.

+ Largest ecosystem, most developers know JS
+ Easiest content pipeline: JSON + React components
+ Monaco editor for in-game "log" displays
− ~180 MB binary (Chromium + Node bundled)
− Worst memory footprint of all options
~ Steam: electron-builder + greenworks native module
Key Tech
electron react greenworks ink

KEY SYSTEM DESIGNS

[SYS-01] Narrative Engine — text templating & incident reports

The game generates narrative text by filling templates with runtime state. Each scientist has a voice (tone + vocabulary), each anomaly has behavioral descriptors, each experiment has outcome phrases.

Template Structure
{scientist.name} {experiment.verb_past} the {anomaly.descriptor} entity. Casualty report: {outcome.personnel_text}. Technology acquired: {outcome.tech_text}. [INTEGRITY: {scientist.integrity_display}]

Templates live in JSON/TOML files — easy to author, version, and mod.

Scientist Voice

Each scientist has a tone enum: CLINICAL, RECKLESS, PHILOSOPHICAL, PARANOID, RESIGNED. This selects from different template banks. A RECKLESS scientist reports casualties with excitement. A RESIGNED one with ellipses.

CLINICAL RECKLESS RESIGNED PARANOID
[SYS-02] ASCII Art Renderer — scientists, anomalies, UI chrome

Every scientist and anomaly has a unique multi-line ASCII art block stored as a text asset. The renderer displays these with the scientist's primary color tinting all associated text.

DR. VASQUEZ ┌─────────┐ │ ★ ★ │ │ ω │ │ ╔═══╗ │ └──┤ A ├──┘ └───┘ INTEGRITY: 74%
DR. █████ ┌─────────┐ │ ■ ■ │ │ ? │ │ ╔═══╗ │ └──┤ ■ ├──┘ └───┘ INTEGRITY: ??%
Art State System

Scientists degrade visually. A scientist at 80% integrity looks normal. At 50%: partial corruption in the art (some chars replaced with glitch chars). At 20%: heavily corrupted, color desaturated. This is the hidden resource made partially visible. Art is stored as multiple versions (INTACT, DAMAGED, CORRUPTED) per character.

[SYS-03] Hidden Resource Tracker — the game's true ledger

Resources are tracked invisibly. The player is never shown a number. They infer state from environmental signals — word choice in reports, visual degradation of scientists, increasingly strange anomaly descriptions.

TECH CORPUS ███░░░░░░░ 31/100
ACTIVE PERSONNEL 6/12 scientists viable
ANOMALY PRESSURE ███████░░░ 71/100

Hover the redacted values above to "peek." Players never see these.

Signal Design
↓ Humanity: Reports become clinical, not personal. Scientist names replaced with IDs.
↑ Tech: New experiment options unlock. Descriptions hint at "harvested" knowledge.
↓ Personnel: Fewer scientists available. Art shows corruption.
↑ Pressure: Anomalies arrive faster, descriptions grow more urgent.
[SYS-04] Content Pipeline — anomaly database & moddability

All content (anomalies, scientists, experiments, containment methods, outcome text) lives in structured data files. This enables non-code content updates, community modding, and rapid design iteration.

NEXT DECISIONS — HIGH LEVEL

With stack locked, these are the design decisions that unblock actual implementation. Resolve in roughly this order — each one gates the next.

DEC-01 Run structure: Roguelike — confirmed

Fresh run each time, anomalies drawn from the hand-crafted pool in a randomized order. Each run is a complete arc — starting with a functioning foundation, ending at an end condition. This means fewer anomalies are needed to feel complete at launch, and replayability comes from variable ordering and different scientist assignments rather than scripted plot. Every run ends at roughly the same thematic place, but through different costs.

RESOLVED
DEC-02 Scientist lifecycle: all three fates — confirmed

Scientists can experience all three outcomes, driven by anomaly type and outcome rolls:

DEATH
Removed from roster permanently. Art slot goes dark. The most brutal signal — their color disappears from the log forever.
TRANSFORMATION
Scientist becomes something else — possibly a new anomaly entry, possibly a resource. The most dramatic outcome; used sparingly for specific anomaly types.
INCAPACITATION
Unavailable for N rounds. Art shows damage states. The most common bad outcome — a warning, not a finality.

Implementation note: each scientist needs art assets for INTACT / DAMAGED / CORRUPTED / ABSENT states. Transformation requires an optional anomaly_entry field on the scientist record.

RESOLVED
DEC-03 Outcome model: probabilistic — confirmed

Outcomes roll dice, modified by scientist stats, anomaly threat level, and hidden resource state. The player sees qualitative risk signals ("this experiment is high-risk", "Dr. Morrow is not suited for this") but not the actual numbers. The hidden ledger adjusts the rolls invisibly — a low-humanity run has worse outcomes at the same nominal probabilities, but the player doesn't see the modifier. This gives the "obscured" feel while keeping the mechanics fair: the dice exist, they just have invisible weights.

outcome_roll = rand(0..100) + scientist.modifier + experiment.risk_offset - hidden_humanity_penalty
RESOLVED
DEC-04 End conditions: 3 endings — confirmed
Containment Failure BAD

Something breaches. Anomaly Pressure maxes out — you ran out of time or made too many mistakes. Catastrophic, sudden, final. The last incident report is filed by no one.

trigger: anomaly_pressure ≥ 100
Transformation DARK

The Director — you — are transformed into an anomaly. Humanity reaches zero. The foundation endures without its soul. The final screen is your own anomaly file, written in clinical third person.

trigger: humanity ≤ 0
Full Containment HOLLOW

All anomalies contained. The foundation stands. A "good" ending — but the accumulated cost of every decision is laid bare in the final report. The player reads back through what they did.

trigger: all anomalies resolved, pressure < 100, humanity > 0
RESOLVED
DEC-05 Screen layout: single scrollable log — confirmed

The entire game lives in one main screen: a scrollable terminal log. ASCII art appears inline when you interact with something — selecting a scientist renders their art into the log, hovering over an anomaly pops its art. Text and art are interwoven in the scroll history. This model means:

[ROUND 7 — BRIEFING] ╔═══════════════════╗ ║ ANOMALY-048 ║ ║ /\ ___ /\ ║ ║ ( ◈ ___ ◈ ) ║ ║ \ [===] / ║ ║ ▓▓▓▓▓▓▓▓▓ ║ ╚═══════════════════╝ Classification: KETER Threat: ████████░░ > Assign lead scientist: [ Dr. MORROW ] ← [ Dr. VASQUEZ ] [ Dr. ████████ ]

Textual implementation: A single RichLog or custom ScrollableContainer widget that renders both text and static art panels inline. Interactive selections are rendered as focusable widgets injected into the log stream.

History scroll: Completed rounds stay visible above — the player can scroll back and re-read all incident reports. The accumulated history IS the game's atmosphere.

Art sizing: All ASCII art must fit within a fixed column width (e.g., 40–50 chars) so it renders cleanly in the log without breaking the scroll column.
