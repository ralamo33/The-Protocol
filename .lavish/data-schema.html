Entity Field Reference

Each field, its type, and what it makes the player feel.

containment_skill int 1–10 roll Adds skill × 2 to resolution roll. High containment = higher success chance on dangerous experiments.
research_skill int 1–10 gate Unlocks additional experiment flavor text and analysis lines in the log. Potential future: unlocks extra intel on anomaly.
risk_tolerance int 1–10 roll Bonus to roll on HIGH-risk experiments specifically. The reckless scientist shines in desperate situations.

roll_modifier int ± roll Hidden. Added to the rolled value. Negative = punishing even with good scientists. Positive = forgiving even with weak teams.

pressure_delta int (neg success, pos fail) pacingending Success reduces pressure; failure increases it. Primary pacing lever. Failure cascades make pressure spiral.

requires_tech int gate Locks advanced containment behind tech investment. Cheaper options are always available but weaker on pressure reduction.

Open Design Choices

Four decisions not yet locked. Each has two concrete options with tradeoffs.

CHOICE 01 Scientist Skill System Complexity

How fine-grained should scientist skill data be? This determines how distinct scientists feel and how much player skill knowledge matters.

Option A — Three Separate Skills

As planned: containment, research, risk_tolerance each 1–10.

  • Each scientist feels mechanically distinct
  • Roll formula uses containment + risk_tolerance separately
  • Research skill needs a defined use or it's dead data
  • More content to author per scientist

Option B — Single "Competence" Score

One competence int 1–10, differentiated by tone and signature experiment only.

  • Simpler TOML, fewer dead fields
  • Scientists feel more alike mechanically
  • Tone + color do more narrative differentiation work
  • Easier to balance with MVP scientist count of 3
Lean: Keep Option A but give research_skill a clear purpose now (e.g. adds 1 narrative line per 3 pts) or cut it. Don't ship dead fields.
CHOICE 02 Pressure: Global vs Per-Anomaly

Does pressure exist as one global ledger value, or does each anomaly track its own pressure independently?

Option A — Global Pressure (Current Plan)

One anomaly_pressure int in GameState. All anomalies contribute to and drain from it.

  • Simpler — one number to manage
  • Any anomaly resolution reduces total threat
  • KETER anomalies create disproportionate spikes
  • Past failures accumulate viscerally into one dread counter

Option B — Per-Anomaly Pressure

Each anomaly has its own pressure counter. Global pressure = sum of all active anomaly pressures.

  • Richer model — you can see which anomaly is critical
  • Enables "triage" gameplay (which do I contain first?)
  • MVP only has 1 active anomaly at a time — moot until multi-anomaly
  • More complex save state
Keep Option A for MVP. Per-anomaly is only meaningful if multiple anomalies are active simultaneously — that's a post-MVP feature.
CHOICE 03 Scientist Fate Roll — How Many Outcomes?

When an experiment failure triggers a personnel incident, how is the outcome determined?

Option A — Flat Fate Table (Current Plan)

Three possible fates: DEATH, INCAPACITATION, TRANSFORMATION. Each anomaly could weight these differently.

  • Simple to implement and balance
  • Each fate has clear mechanical consequence
  • Needs per-anomaly fate weight table to feel interesting
  • 3 scientists × 3 fates is enough story surface for MVP

Option B — Severity-Graduated Fate

A 0–100 roll determines severity. Low roll = minor injury (–20 integrity). Mid = incapacitation. High = death/transform.

  • More granular — partial damage feels less binary
  • Adds another hidden number to track and tune
  • Integrity already represents partial damage — overlap risk
  • May soften dramatic scientist deaths if minor hits are common
Option A. Integrity already handles gradual damage; flat fatal fates preserve dramatic weight. Don't dilute death with minor papercuts.
CHOICE 04 Tech Level — How is Progress Signaled?

Tech level is hidden, but players need to understand that investment unlocks options. How do we signal progress?

Option A — Unlock Notification in Log

When tech crosses a threshold, a log line appears: "RESEARCH ARCHIVE UPDATED. New protocols available."

  • Clear, unambiguous signal to the player
  • Fits the terminal-log aesthetic perfectly
  • Risk: player treats it as a progress bar substitute
  • Easy to implement — just compare tech before/after apply_outcome

Option B — Silent Discovery

No notification. New experiments and containment simply appear in the next round's option list.

  • More mysterious — player discovers, doesn't receive
  • Fits the "infer from environment" design ethos
  • Risk: players miss unlocks and feel the list doesn't change
  • Works better if new options have visually distinct presentation
Option B with visual distinction — newly unlocked items get a brief highlight marker on first appearance. Discovery, not notification. Matches the game's hidden-state philosophy.
